agentic-vl System Architecture and Implementation Overview

This document describes the current prototype architecture: a text-first coding agent reads static vvvv projects, observes live vvvv editor state, and sends narrow write requests through an MCP stdio adapter into an HDE-hosted vvvv bridge.

Project: agentic-vl Generated: 2026-06-24 Verified baseline: vvvv gamma 7.2 Read layer stable, write layer experimental

Executive Summary

The system is split into a stable read-only analysis layer and a narrow live-edit layer. Static analysis is handled by vl-map, exposed to agents through vl-mcp. Live editor context is produced inside vvvv by VL.Agent, primarily through the AgentHost node running in VL.Agent.HDE.vl. Live writes are not direct MCP calls into vvvv memory; they are JSON request files written to .agent/requests, consumed by CommandProcessor on the vvvv main loop, and answered through .agent/results.

MCP is intentionally an adapter, not the internal architecture. The transport between the MCP client and vl-mcp is newline-delimited JSON-RPC 2.0 over stdio. The transport between vl-mcp and the in-vvvv bridge is filesystem JSON. The transport inside vvvv uses public vvvv APIs such as VL.HDE.API, SessionNodes.CurrentSolution, DevEnvHost.Instance.LatestCompilation, and immutable VL.Model update operations.

Top-Level Architecture

Components

tools/vl-map

Static project cartographer. It walks a project directory, skips generated folders such as .git, bin, obj, and .vs, parses .vl XML with XDocument, and returns a structured ProjectIndex.

It extracts document IDs, language/version metadata, NuGet and document dependencies, definitions, referenced libraries, duplicate element IDs, node/pad/link/canvas/pin counts, and a vertex-level patch graph.

tools/vl-mcp

MCP server and adapter. It reads JSON-RPC messages line by line from stdin and writes JSON-RPC responses to stdout. Tool and resource descriptions are generated in-process, and larger context is exposed as resources rather than stuffed into tool descriptions.

It calls VlMap.Indexer.Build directly for static queries and uses atomic request-file writes plus result polling for live vvvv commands.

VL.Agent

C# node library loaded into vvvv, targeting net8.0-windows and referencing the installed gamma assemblies directly. Initialization.cs uses [assembly: ImportAsIs(Namespace = "VL.Agent", Category = "Agent")], making public types appear as vvvv nodes.

The preferred host is VL.Agent.HDE.vl, so the bridge runs in the editor runtime instead of the user patch runtime.

Protocols and Transports

Boundary Protocol / Transport Payload Why it is shaped this way
Agent to vl-mcp Newline-delimited JSON-RPC 2.0 over stdio MCP methods: initialize, tools/list, tools/call, resources/list, resources/read, ping Standard MCP integration. stdout must be protocol-clean; diagnostics go to stderr.
vl-mcp to vl-map In-process .NET method call Indexer.Build(projectPath) returns ProjectIndex Static analysis is pure and does not require vvvv to run.
vl-mcp to VL.Agent Filesystem mailbox JSON request files in .agent/requests; JSON result files in .agent/results Crosses the process/runtime boundary without hosting an HTTP server in vvvv. The bridge consumes requests on the vvvv frame loop.
VL.Agent to external agent Filesystem snapshot .agent/editor-state.json Live state can be read by any MCP client process without direct references to vvvv assemblies.
VL.Agent to vvvv editor Public vvvv .NET APIs VL.HDE.API, SessionNodes, DevEnvHost, VL.Model Uses vvvv's editor/runtime surfaces and undo-integrated solution model instead of hand-editing XML.

How Tools Are Called

The MCP client calls tools/call with a tool name and arguments. Tools.Call dispatches by name and always returns MCP text content. Tool implementations fall into three groups: static reads, live snapshot reads, and live requests.

Static read tools

  • vvvv_index_project calls Indexer.Build(projectPath) and returns the full JSON index.
  • vvvv_context_query with projectGraph, patchGraph, or nodeContext also builds a static index, then returns a compact slice.
  • Static graph slices include document edges, packages, patch nodes, pads, links, and neighbor context around one node or pad.

Live read tools

  • vvvv_editor_state reads the raw snapshot file and prefixes path/age comments.
  • vvvv_context_query with summary, documents, selection, compilerMessages, or raw parses editor-state.json and returns the requested slice.
  • agentic-vl://editor/state exposes the latest snapshot as an MCP resource.

Agent Context Management

Context is deliberately tiered so an agent can start compact and pull detail only when needed. This matters because full vvvv project graphs and raw editor snapshots can be large, while most decisions only need loaded documents, current selection, compiler messages, and nearby graph context.

Static Context

vl-map builds context from files on disk. It parses .vl XML by local element name, so XML namespaces do not break extraction. It builds a project-level inventory and a patch-level dataflow graph by collecting nodes, pads, pins, and links, then mapping pin endpoints to owning vertices.

The MCP-facing form is vvvv_context_query: projectGraph for package/document relationships, patchGraph for one document's nodes/pads/links, and nodeContext for inbound/outbound neighbors around a specific node or pad.

Live Context

EditorWatcher mirrors live editor state into editor-state.json. It reads VL.HDE.API.LoadedDocuments, API.CurrentSelection, and latest compiler messages, then serializes a JSON snapshot.

For selected ILiveElement values, the snapshot includes type, base element ID, document ID, parseable UniqueId, element name, symbol, kind, unused flag, element messages, selected node pins, and selected pad/data hub values.

Freshness and Change Detection

EditorWatcher.Update computes a cheap signature from selection identities, selected live values, loaded document paths/dirty flags, and compiler-message count. If the signature and target path are unchanged, it does not rewrite the snapshot.

After a command applies, AgentHost calls ForceWrite immediately and for two additional frames. This helps the external agent observe the result of a write without guessing when vvvv's live model settled.

vvvv Integration

vvvv gamma is a live, frame-based .NET visual programming environment. Patch edits, shader edits, and source-project C# changes can be reflected while the program is running. This repo leans on that model but keeps the bridge in an HDE/editor extension where possible, so the control plane is separated from the patch being edited.

How VL.Agent Enters vvvv

  • VL.Agent.csproj targets net8.0-windows, matching the vvvv runtime.
  • It references VL.Core.dll, VL.Lang.dll, VL.AppServices.dll, VL.HDE.dll, and System.Reactive.dll from the installed vvvv gamma 7.2 directory.
  • Initialization.cs imports the namespace as vvvv nodes in category Agent.
  • AgentHost is a [ProcessNode], so it has vvvv's constructor/update lifecycle and runs every frame while enabled.

AgentHost

AgentHost composes EditorWatcher and CommandProcessor. It resolves the agent directory from loaded non-HDE documents, preferring changed project documents, and falls back to an explicit path input. Each update writes or refreshes the editor snapshot, drains pending commands, and forces extra snapshots after successful commands.

EditorWatcher and EditorBridge

EditorBridge.WriteEditorSnapshot uses VL.HDE.API, which is globally reachable inside the editor runtime. It snapshots loaded documents, current selection, and compiler messages to JSON. Selection objects are inspected as ILiveElement, ILiveNodeApplication, or ILiveDataHub when possible.

CommandProcessor

CommandProcessor.Update looks for *.json files under .agent/requests, sorts them ordinally, processes each, writes a result file with the same filename under .agent/results, deletes the request, and updates a status counter. Supported request ops are setPinValue, openDocument, nodeQuery, graphTransaction, and paste.

Live Mutation Strategy

Simple pin edits call SessionNodes.CurrentSolution.SetPinValue(uid, pin, value) and then Confirm(SetPinUpdateKind). Graph transactions validate all operations first where possible, collect diagnostics, and then apply supported steps using a mix of CurrentSolution, ModelExtensions, Patch.AddPad, ModelExtensions.AddNode, and Patch.GetOrAddLink.

Safety boundary: direct paste from CommandProcessor.Update() was observed to race the Skia patch editor render loop and throw InvalidOperationException: Collection was modified; enumeration operation may not execute. The current paste path is explicitly dev-only, requires experimental=true, posts to the UI synchronization context, and can temporarily pause the user runtime.

Graph Transactions

A graph transaction is a schema-versioned JSON batch with a human label, optional dryRun and validate flags, and an ops array. It is intended to be one logical user-visible operation, though full atomicity is not proven for all structural edits yet.

Operation Current support Important constraints
setPin Supported Targets <UniqueId>:<PinName> or transaction-created aliases as <alias>:<PinName>. Batched live node pin edits can be one confirm.
addNode Experimental Creates operation/process nodes on the active canvas. Symbols are resolved against live compilation. Use vvvv_node_query first.
addPad Experimental Creates primitive typed IOBoxes/pads on the active canvas.
connect Experimental Supports aliases, pad endpoints, and node-pin endpoints. Created-node endpoints are checked for pin direction and type.
setBounds First slice Moves node/pad targets by full UniqueId, resolving current selection first and then the live graph model.
select First slice Can select transaction-created aliases or existing node/pad UniqueId targets from the live graph model.
validate Supported diagnostics hook Validation results are included in the transaction result.
disconnect, annotate Schema exists, implementation unsupported Reported as unsupported until a safe path is proven.

Path Conventions

<project>/.agent/editor-state.json
<project>/.agent/requests/<guid>.json
<project>/.agent/results/<guid>.json

vl-mcp uses its working directory as the project root. AgentPaths.DefaultStatePath can be overridden with VVVV_AGENT_STATE. Tool calls that submit live requests can override the agent directory with agentDir. Some target-resolution paths accept projectPath, documentPath, and documentId so a bare element ID can be expanded to a full vvvv UniqueId.

Current Risks and Open Edges

  • Structural graph edits are still experimental. Some late failures can leave earlier structural changes committed; results report partial:true when this is detected.
  • Paste is dev-only and should not be treated as stable MCP graph insertion.
  • setBounds and select now share model lookup with connect, but still need Windows runtime verification for arbitrary unselected targets.
  • Node insertion depends on the active canvas and the latest live compilation, so a focused patch and healthy vvvv compile state are prerequisites.
  • The stable, recommended loop is observe first, use compact context queries, dry-run graph transactions, then apply the smallest viable edit.
