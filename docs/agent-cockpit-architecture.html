agentic-vl / architecture proposal / 2026-06-20

Make vvvv an agent-native live environment.

The next system should not be a chat bolted onto a patcher. It should be an HDE-side cockpit with fast graph transactions, live probes, validation, and a replaceable agent backend.

selected graph -> structured context
agent plan -> transaction preview
apply -> validate -> probe
backend adapter -> swappable runtime

00 / thesis

The HDE extension should become the agent cockpit.

The visual nature of vvvv is an advantage, not a problem to hide behind text. The project tab can stay focused on the user patch, while the HDE extension exposes the agent's context, plan, proposed graph operations, probes, and validation results.

The backend should remain replaceable. Codex, pi, Claude, or a custom local planner can all sit behind the same narrow local protocol. The vvvv-specific value lives in the HDE control plane.

North star

A user asks for behavior, the agent proposes a graph transaction, vvvv previews it, applies it as one undoable operation, watches the result for a few frames, and revises.

01 / bottlenecks

The current loop is correct, but too low-bandwidth.

Mutation is too granular

One request per small edit forces the agent into minutes of node-by-node work. Complex patches need batch graph transactions.

Observation is too thin

The snapshot has documents, selection, and messages, but not enough pin schemas, live values, node catalog data, thumbnails, or graph deltas.

Insertion is not proven safe

Paste exists as an experiment, but safe node/link creation needs an editor-context operation path with validation and undo.

No shared transaction contract

The HDE bridge, MCP server, and future UI need one typed schema for add-node, connect, set-pin, layout, validate, and revert.

No live feedback loop

The agent needs to subscribe to values, channels, compiler messages, runtime exceptions, and frame metrics after each change.

Chat is not contextual enough

The chat should be scoped to active document, selected graph, probes, and proposed operations, not just the repository text.

02 / system shape

Separate the cockpit, bridge core, and agent runtime.

HDE Agent Cockpit

User-facing extension UI inside vvvv.

  • Chat and task panel
  • Graph operation preview
  • Live probe panels
  • Validation and error stream
  • Agent activity timeline

AgentHost Core

Resident editor-side control plane.

  • Graph snapshots and deltas
  • Node catalog and pin schemas
  • Batch graph transactions
  • Runtime probes and channels
  • Undo and validation hooks

Runtime Adapter

Replaceable reasoning backend.

  • Codex adapter
  • pi adapter
  • Claude adapter
  • Scripted test adapter
  • Future custom runtime
03 / phases

Build throughput first, then wrap it in the cockpit.

phase 1

Stabilize the resident bridge

Keep AgentHost as the default and evolve it into a continuous service. Replace file polling with a faster local transport if feasible, while keeping file mode as a fallback.

Open questions: named pipes, TCP, WebSocket, or child process? Can HDE code host this reliably?

phase 2

Export a real graph model

Expose nodes, pads, pins, links, bounds, selection, compiler messages, and graph deltas as structured data. Add a searchable node and pin catalog.

Open questions: which API exposes authoritative node browser and pin schema data?

phase 3

Implement graph transactions

Support batch ops: add node, add pad, set pin, connect, create region, layout, annotate, validate, and select. Return created IDs and diagnostics.

Open questions: safe editor API for add/connect? One undo step per transaction?

phase 4

Add live probes and validation

Read selected values for N frames, subscribe to channels, monitor runtime exceptions, capture compile diagnostics, and validate visual output where feasible.

Open questions: access to non-selected live values? Skia or Stride output capture?

phase 5

Build the HDE cockpit UI

Show the chat, current context, proposed transactions, activity timeline, probe panels, and apply/revert controls inside vvvv.

Open questions: native VL/Skia, ImGui, custom window, or embedded web UI?

phase 6

Abstract the agent backend

Define a local protocol so the cockpit can talk to Codex, pi, Claude, a test adapter, or a future purpose-built vvvv agent runtime.

Open questions: which external agent protocol is stable enough to wrap first?

05 / benchmarks

Use real scenarios as throughput tests.

Particle system

The agent should scaffold a dense simulation core in C# or Fuse/Stride, then patch the control surface.

  • Emitter parameters and lifetime
  • Physics and forces
  • Collision settings
  • Rendering mode and debug views
  • Validation: count, bounds, frame time, exceptions

Skia plus OSC

The agent should generate OSC routing, typed channels, a Skia renderer patch, and mock input for validation.

  • OSC address map
  • Channel schema
  • Skia layout and draw layers
  • Mock OSC sender
  • Validation: channel updates and visible output changes
06 / first move

Start with the contract before building the cockpit.

The first concrete step is a GraphTransaction schema. It lets the MCP tool, HDE bridge, and future UI agree on what an agent is allowed to propose, preview, apply, validate, and undo.
