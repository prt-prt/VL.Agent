1. Executive summary

vvvv gamma is best understood as a live .NET development environment where visual patches, C# source code, shaders, packages, and runtime state coexist inside a continuously running application. It is not merely a node editor, and not merely a C# host. It is a compiler/runtime/editor/package ecosystem with strong emphasis on real-time feedback.

The central workflow is: users author .vl XML documents visually; those documents reference packages, other documents, C# source projects, compiled assemblies, and shaders; vvvv resolves those dependencies, compiles visual programs and C# code, creates runtime node instances, and evaluates connected graphs every frame.

For agentic development, the key insight is that vvvv has multiple partially overlapping representations of a project:

  • File representation: XML documents, C# files, shader files, project files, NuGet package specs.
  • Editor representation: selected/hovered nodes, patch canvas, node browser, inspector, command system.
  • Compiler representation: resolved symbols, type definitions, generated C#, target compilation, diagnostics.
  • Runtime representation: process node instances, frame clock, channels, resources, GPU state, exceptions, logs.

A capable vvvv-agent must eventually bridge these layers instead of treating .vl as plain XML or C# as plain code.

What is unusually powerful

  • Live visual editing while the program runs.
  • Source C# projects can be compiled by vvvv via Roslyn and hot-reloaded.
  • Shaders live-reload.
  • Any .NET library can be wrapped or imported as nodes.
  • Packages combine visual patches, code, shaders, docs, and tests.

What is unusually complex

  • The same concept can exist as XML, symbol, generated code, node browser entry, and runtime instance.
  • Visual graph behaviour depends on frame timing, stateful nodes, spreading, nil handling, and resource lifecycles.
  • Patch mutation through raw XML is possible but semantically fragile without editor/compiler feedback.

3. Documents and file model

The .vl document is the core project artifact. It is XML-based and version-controlable. It stores visual definitions, dependencies, patches, nodes, pins, pads, links, canvas layout, regions, type annotations, and process lifecycle fragments.

3.1 Root structure

<Document xmlns:p="property" xmlns:r="reflection"
          Id="..." LanguageVersion="2024.6.0" Version="0.128">
  <NugetDependency Location="VL.CoreLib" Version="..." />
  <DocumentDependency Location="./Other.vl" />
  <PlatformDependency Location="VL.Core.dll" />
  <Patch>...</Patch>
</Document>

Dependencies are direct children of Document, not nested inside patches. Almost every document needs VL.CoreLib. IDs are unique 22-character base62 GUID encodings.

3.2 Important element types

ElementRoleAgent-relevant notes
DocumentWhole file and dependency container.Language version, namespace declarations, dependency list.
PatchLogical container for graph fragments and definitions.Can represent top-level, Create/Update fragments, region bodies.
CanvasVisual grouping/layout.Mostly presentation, but important for human readability.
NodeOperation call, process definition, region, or other symbol reference.Meaning is primarily in p:NodeReference.
PinInput/output endpoint.Links reference pin IDs; pin kind matters.
PadIOBox/value/property visual node.Often used for constants, comments, patch inputs/outputs.
LinkConnection between output and input.Ids="sourcePinId,sinkPinId", direction matters.
ProcessDefinitionLifecycle definition.References sibling Create/Update patches through fragments.
SlotState field in a type definition.Represents persistent state inside visual types.

3.3 NodeReference and Choices

A node's visible Name is not enough to know what it does. Its semantic identity is encoded in p:NodeReference choices. Examples include operation calls, process applications, type definitions, and stateful regions.

<p:NodeReference LastCategoryFullName="Primitive.Math" LastDependency="CoreLibBasics.vl">
  <Choice Kind="NodeFlag" Name="Node" Fixed="true" />
  <Choice Kind="OperationCallFlag" Name="+" />
</p:NodeReference>

This means a static XML analyzer can extract useful information, but full correctness requires resolving choices against the loaded package/document symbol universe.

3.4 Regions

Regions are visual control-flow constructs such as If, ForEach, Cache, Repeat, and Accumulator. In XML, they are nodes with stateful-region choices and nested patches/fragments. They are not just visual boxes: they alter evaluation and scope.

Analysis consequence: arbitrary XML editing is risky. A valid-looking node/link/pin edit can still be semantically invalid if the node reference does not resolve, if fragment references are wrong, or if type inference fails.

4. Editor / HDE layer

The editor, often referred to as HDE in vvvv terminology, is where patchers work visually. It provides the canvas, node browser, inspector, menus, windows, commands, and selection state.

4.1 Editor extensions

Editor extensions are ordinary VL patches with a .HDE.vl suffix. They run automatically when open in the editor and can add commands and windows.

MechanismPurpose
VL.HDEProvides command nodes, editor window types, docking helpers.
VL.LangProvides Session API nodes for editor/compiler/project interaction.
Command nodeRegisters a menu command, optionally with shortcut.
SkiaWindow / SkiaWindowTopMostCustom extension UI surfaces.
WindowFactoryDocking integration for extension windows.

4.2 Why this matters

The editor has context that files do not: current selection, hovered node, visible patch, UI state, inspector values, maybe resolved symbol identities. For an agent, an HDE bridge is likely the most feasible way to become editor-aware without reverse-engineering the entire compiler model.

Missing detail: this report does not include a full inventory of VL.Lang Session nodes or their exact data types. That needs a Windows/vvvv session or direct current gamma API documentation/source. The available information says selected/hovered nodes and pin read/write APIs exist, but not their complete shape.

5. Runtime and evaluation model

vvvv gamma evaluates connected dataflow graphs frame by frame, typically around display refresh rates. The runtime has to maintain process node state, manage resources, update compilation results, and preserve live-programming continuity.

5.1 Frame-based execution

  • Data generally flows left-to-right, top-to-bottom.
  • Connected graph parts evaluate every frame.
  • Disconnected subgraphs are skipped.
  • Stateful process nodes keep state across frames.
  • Frame delay nodes break feedback cycles by introducing one-frame latency.

5.2 Process lifecycle

ConceptVisual patchC# node
CreateCreate fragment/region runs once for instance setup.Constructor runs once per node instance.
UpdateUpdate fragment runs each frame.Update(...) is called each frame.
DisposeImplicit/explicit cleanup depending on resources.IDisposable.Dispose() called on removal/reload/document close.

5.3 Hot reload

Patch changes apply live and generally preserve node state. C# source project changes trigger recompilation and lifecycle restart of affected nodes: old instance is disposed, a new constructor runs, and Update resumes on the next frame. Static fields reset because the in-memory assembly is replaced.

5.4 Runtime resources

Real vvvv projects frequently manage resources that outlive a single frame: GPU buffers/textures, Stride game systems, native handles, network sockets, device connections, subscriptions, and C# services. Resource management is therefore a first-class concern. Missing disposal causes leaks across both runtime operation and live reload cycles.

Agent angle: runtime behaviour cannot be inferred from static graph structure alone. A useful debugging assistant needs logs, exceptions, frame events, channel/probe values, and knowledge of lifecycle semantics.

6. Nodes, type system, and graph semantics

6.1 Node categories

KindDescriptionState?
Operation nodePure or mostly stateless function call; same input should produce same output.No persistent instance state.
Process nodeStateful node with lifecycle and per-frame update.Yes.
Adaptive nodeResolves implementation from connected types, e.g. arithmetic operators.Depends on resolved implementation.
RegionControl-flow construct such as If/ForEach/Cache.Often stateful.
Pad / IOBoxValue holder, property access, patch boundary, or comment.Can represent persistent/editable values.

6.2 Spreads

Spread<T> is vvvv's immutable collection type. It is central to dataflow programming because many nodes automatically operate over spreads. In visual patching, indexing often uses cyclic semantics; in C# code, authors must handle count and indexing explicitly.

Performance-sensitive C# nodes should avoid LINQ and per-frame allocations. Use SpreadBuilder<T>, cache unchanged outputs, and prefer ReadOnlySpan<T> internally or for hot-path outputs where appropriate.

6.3 Nil and empty data

vvvv workflows often need to distinguish absent/empty data from valid data. In C#, spreads should never be null; use Spread<T>.Empty. Ignoring nil/empty cases is a common source of visual-patch bugs.

6.4 Event nodes and temporal semantics

Common temporal nodes/patterns include:

  • Bang: one-frame trigger.
  • Toggle: persistent boolean state toggled by events.
  • Changed: detects value changes between frames.
  • FrameDelay: delays a value by one frame and breaks cycles.

These are crucial because vvvv programs are temporal systems, not just data transformations.

7. .NET and C# integration

vvvv gamma is deeply integrated with .NET 8. C# can be used to implement custom nodes, wrap libraries, manage resources, and expose services. This makes the .NET ecosystem available from visual patches.

7.1 Source project vs binary reference

WorkflowHow it worksProsCons
Source project reference.vl references a .csproj; vvvv compiles source via Roslyn into in-memory assemblies.Live reload, fast iteration, no external build needed during patching.Agent cannot directly see vvvv compiler output unless bridged; state resets on reload.
Binary / NuGet referencevvvv loads pre-built DLL/package.Stable, package-friendly, suitable for larger libraries.Requires external build and vvvv restart to pick up changes.

7.2 Import model

Types appear in the node browser when they are public and covered by an assembly-level import attribute:

[assembly: ImportAsIs(Namespace = "MyCompany.MyLibrary", Category = "MyLibrary")]
// or ImportNamespace(...) multiple times
// or ImportType(typeof(MyNode), Category = "...")

[ProcessNode] does not make a class visible. It marks it as a stateful lifecycle-managed node. Visibility is controlled by public API plus import attributes.

7.3 ProcessNode C# pattern

[ProcessNode]
public sealed class MyProcessor : IDisposable
{
    private float _lastInput;
    private float _cached;

    public void Update(out float result, float input = 0f, bool reset = false)
    {
        if (reset || input != _lastInput)
        {
            _cached = Compute(input);
            _lastInput = input;
        }
        result = _cached; // always assign outputs
    }

    public void Dispose() { /* release resources */ }
}

Key conventions: out parameters first, value inputs after, no unnecessary public helpers, no Node suffix in user-facing names, XML comments for tooltips, dispose native resources, avoid allocations in Update.

7.4 Services and NodeContext

Process node constructors can receive NodeContext to access app-host services, frame clocks, Stride game context, logging, or package-registered services. Node libraries can register services through AssemblyInitializer.

7.5 Async and threading

Update() runs on the VL main thread, so long-running work should be moved to tasks or background threads. Background results should be marshalled back through the captured SynchronizationContext.

8. Packages, libraries, and dependency resolution

vvvv functionality is distributed through packages, commonly NuGet packages that can contain a main .vl document, compiled DLLs, shaders, help patches, tests, and metadata.

8.1 Package layout

VL.MyLibrary/
  VL.MyLibrary.vl
  VL.MyLibrary.nuspec
  lib/net8.0/VL.MyLibrary.dll
  src/VL.MyLibrary.csproj
  shaders/MyEffect_TextureFX.sdsl
  help/HowTo Use MyNode.vl
  tests/MyFeatureTest.vl

A library is recognized when folder name, main .vl file, and .nuspec share the same name. C# projects should compile into lib/net8.0/ for package deployment.

8.2 Dependency types

DependencyMeaning
NugetDependencyReference to a NuGet/VL package such as VL.CoreLib or VL.Stride.
DocumentDependencyReference to another .vl document.
PlatformDependencyReference to platform assemblies such as DLLs.
PackageReference in .csprojNuGet dependency for C# source projects.

8.3 Editable packages

For development, vvvv can load local packages from source using CLI flags:

vvvv.exe --package-repositories "D:\Projects" \
         --editable-packages "VL.MyLib*" \
         -o "D:\Projects\VL.MyLib\help\HowTo Use Feature.vl"

--package-repositories points at parent folders containing packages; --editable-packages chooses which packages to load from source.

Pitfall: pointing package repositories at a folder containing large standard-library submodules can cause vvvv to discover and recompile a lot of source, slowing startup dramatically.

9. Channels and reactive state

Channels are named, typed, observable value containers. They provide a reactive state layer that can be accessed from patches, C# nodes, UI bindings, and external systems.

9.1 Concepts

ConceptDescription
IChannel<T>Observable container with current value.
IChannelHubApp-wide hub for public channel lookup.
PathDot-separated hierarchical name, e.g. Project.Scene.Camera.Position.
[CanBePublished(true)]Marks .NET types whose public properties can become sub-channels.
Bang channelEvent-like channel, usually System.Reactive.Unit.

9.2 Hierarchical propagation

When a publishable object is exposed, vvvv can create sub-channels for properties. Writing a root model updates children; writing a leaf can propagate upward. Spread elements can be addressed with bracket paths such as Project.Items[0].Name.

9.3 C# access pattern

var hub = IChannelHub.HubForApp;
var channel = hub.TryGetChannel("Settings.Audio.Volume");
if (channel?.Object is float volume)
    Apply(volume);

Nodes should retry-bind because channels may not exist at constructor time. Subscriptions must be disposed. Feedback loops require suppression flags when writing to a channel you also observe.

Critical rule from skill corpus: avoid TryAddChannel() for public channel creation because it can create null-value channels that crash sub-channel reflection. Use lookup-only patterns and let the publishing system create channels.

10. Rendering, shaders, and media ecosystem

vvvv gamma uses Stride for 3D rendering and SDSL for shaders, while also supporting 2D rendering through packages such as VL.Skia and GPU/compute workflows through packages such as VL.Fuse.

10.1 Stride and SDSL

SDSL is Stride's shader language: a superset of HLSL with shader classes, inheritance/mixins, streams, composition, and override semantics. Shaders live in .sdsl files and can be auto-registered as nodes by filename suffix.

SuffixGenerated node categoryTypical use
_TextureFX.sdslTextureFXImage processing/filter passes.
_DrawFX.sdslDrawFXDrawing/rendering shaders.
_ComputeFX.sdslComputeFXGPU compute workloads.
_ShaderFX.sdslShaderFXComposable shader graph effects.

10.2 Streams and composition

SDSL streams avoid manual VS/PS struct plumbing. ShaderFX composition lets patches connect shader components visually, while SDSL compose slots provide dynamic shader composition.

11. Testing, debugging, and diagnostics

11.1 VL.TestFramework

Package authors can use VL.TestFramework with NUnit to load and compile .vl documents and optionally run entry points.

[Test]
public async Task MyPatchCompilesAndRuns()
{
    await testEnvironment.LoadAndTestAsync("path/to/MyPatch.vl", runEntryPoint: true);
}

Test documents commonly live in tests/; help patches in help/ can be compile-tested. Test nodes in VL patches can assert equality, non-null values, expected errors, and sequence equality.

11.2 Debugging C# nodes

Debugging usually involves launching or attaching to vvvv.exe from Visual Studio or VS Code. The --debug flag improves debug symbol support but slows live compilation. --stoppedonstartup helps set breakpoints before updates start. --nocache can help with stale symbols.

11.3 Logs

When enabled, vvvv writes logs to %USERPROFILE%\Documents\vvvv\gamma\vvvv.log. Logs are crucial for agentic diagnostics because they expose compile/runtime errors that static analysis cannot see.

11.4 Common failure classes

AreaTypical issueLikely fix
C# node visibilityNode not in browser.Check import attributes, public API, target net8.0, build errors.
Runtime stateNull/default downstream outputs.Always assign outputs, cache values, handle nil/empty spreads.
PerformanceFrame drops / GC spikes.Avoid allocations and LINQ in Update, cache resources.
ThreadingIntermittent crashes.Marshal background results to VL thread.
CyclesCircular dependency warnings.Insert FrameDelay.
ShadersSDSL compile errors.Check semicolons, override, static const placement, enum binding.

13. Observations from the public vvvv-sdk clone

The repository cloned into this project at vvvv-sdk/ appears primarily to be historical vvvv45/public SDK code, not current vvvv gamma source. It therefore should not be treated as authoritative for gamma internals. However, it provides useful architectural signals.

13.1 Runtime host pattern

vvvv45/src/integration/VVVV.VLIntegration/src/RuntimeHost.cs shows a host that manages runtime instances, frame events, compilation updates, exceptions, run modes, and hot-swap synchronization. This reinforces the idea that a live VL system has observable runtime events: frames, updates, and exceptions.

13.2 Node factory pattern

NodeFactory.cs shows .vl integration through a factory that loads documents, discovers node definitions, updates node infos, creates runtime instances, and responds to compilation updates. This matches the broader model of documents being compiled into runtime entry points rather than directly interpreted as static XML.

13.3 Command pattern

The historical core has serializable commands such as ADD, REMOVE, SET, RENAME, and COMPOUND with undo/redo semantics. Even if gamma's current internals differ, this is an excellent model for agentic patch mutation: small typed operations, transaction grouping, undoability, and serialization.

Limit: because current gamma internals were not available locally, exact current APIs for host/session/compiler/command systems remain unverified.

14. Missing information and open research areas

Known gaps

  1. Current gamma source internals: the public clone is historical vvvv45. Current gamma compiler/editor/runtime source was not available in this session.
  2. VL.Lang Session API inventory: exact nodes, capabilities, data types, mutation functions, and limitations need inspection inside vvvv.
  3. Editor command/mutation API: unclear whether external or HDE extension code can safely perform all operations needed for arbitrary patch edits with undo integration.
  4. Node browser/symbol resolution: static XML can identify many references, but full semantic resolution likely requires vvvv's compiler/session.
  5. Package resolver details: exact cache, editable package precedence, version conflict resolution, and NuGet source behaviour are not fully mapped.
  6. Runtime introspection: exact access to live node instances, pin values, errors, timing, and allocation metrics requires experimentation.
  7. VL.Stride/Fuse internals: render graph, GPU resource lifecycle, shader compilation pipeline, and effect composition need package-specific investigation.
  8. Export/deployment pipeline: this report does not analyze exported app build mechanics in depth.

15. Implications for agentic development

A vvvv-agent should be designed around the ecosystem's real layers rather than forcing vvvv into a normal text-code workflow.

High-confidence capabilities

  • Index project files and dependencies.
  • Parse and validate basic .vl XML structure.
  • Generate and review C# node libraries.
  • Set up package structure, .csproj, imports, help/test layout.
  • Parse logs and common errors.
  • Generate VL.TestFramework test scaffolding.

Capabilities needing a bridge

  • Know what the user has selected or hovered.
  • Resolve node references semantically against loaded packages.
  • Read live pin values or runtime errors in context.
  • Apply patch mutations through editor-safe commands.
  • Observe frame timing and runtime resource state.

Recommended conceptual architecture

External agent core ├─ File tools: read/write/search/git/dotnet ├─ vvvv static tools: vl-map, vl-lint, patch-op planner ├─ C# tools: node/library generator, analyzer, tests ├─ Runtime tools: log parser, test runner, launcher └─ Bridge client ├─ HDE extension: selection, hovered node, commands, windows ├─ Channel bridge: debug values, app model, probes └─ Service bridge: deeper package/runtime integration

Practical stance

The safest path is to start with read-only understanding and C# support, then add runtime/test/log feedback, then add editor context, and only later attempt transactional patch mutation. The mutation layer should use a typed operation IR, not raw LLM-authored XML.

Bottom line: vvvv is already a live, multi-representation programming system. Agentic development is feasible if the agent respects those representations: XML files, visual editor context, compiler symbols, runtime instances, channels, packages, and tests.
